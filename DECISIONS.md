# Decisions

本次更新時間：26/06/10 14:26

## 26/06/10 02:56

- 決策：網站正式名稱定為「酒館角色卡製卡工坊」。
- 理由：名稱直接表達用途，符合 SillyTavern 創作者工具定位。

## 26/06/10 02:56

- 決策：第一版只做靜態前端。
- 理由：目前核心需求是本機處理、模板、AI prompt 複製、JSON/PNG 解析與匯出，不需要後端。

## 26/06/10 02:56

- 決策：第一版就支援 PNG 角色卡。
- 理由：SillyTavern 使用者常用 PNG 角色卡；若只支援 JSON，會降低工具實用性。

## 26/06/10 02:56

- 決策：左側步驟導覽固定為「使用說明、角色卡、世界書、合併、匯出」。
- 理由：比「素材 / AI Skill / 角色卡 / 世界書 / 合併 / 匯出」更符合實際工作流，且 AI Skill 應靠近使用情境。

## 26/06/10 02:56

- 決策：角色卡相關 AI Skill 放在角色卡區塊，世界書相關 AI Skill 放在世界書區塊。
- 理由：創作者使用時不需要切到獨立頁找 prompt，能在對應步驟直接複製。

## 26/06/10 02:56

- 決策：範例庫只放三個模板：角色卡空白模板、世界書空白模板、角色卡 + 世界書合併模板。
- 理由：MVU、劇情向、群像卡、世界觀拆條等內容不是固定模板能簡單生成，放進範例庫容易誤導創作者。

## 26/06/10 03:01

- 決策：暫不初始化 Trellis。
- 理由：目前先建立輕量 continuity 文件即可開始；Trellis 屬於長期任務管理工具，等使用者明確確認後再啟用。

## 26/06/10 03:20

- 決策：使用者回覆「開始做吧」後，先沿用目前專案資料夾進行靜態前端實作，暫不建立新 repo。
- 理由：需求是先把 MVP 做起來；但後續若要部署到 GitHub，仍需建立 repo 並提交推送。

## 26/06/10 11:18

- 決策：網站改為兩個工作流：「全新製作」與「修卡」。
- 理由：全新製作與修卡的使用者心智不同；全新製作應直接看到空白模板，修卡則需要匯入、手動修正與拆分 JSON 給 AI 的路徑。

## 26/06/10 11:18

- 決策：修卡工作流提供「拆出角色卡 JSON」與「拆出世界書 JSON」。
- 理由：方便使用者把角色主體與世界書分開交給 GPT / Gemini / Claude 等 AI 修正，再回網站合併匯出。

## 26/06/10 11:18

- 決策：世界書 AI Skill 融合 `世界書擺放與注意力分佈.md`。
- 理由：世界書不是只要格式正確，還需要考慮插入位置、注意力分佈與排序策略，否則 AI 生成的 entries 可能不好用。

## 26/06/10 11:39

- 決策：世界書匯入必須同時支援 `entries` 陣列格式與物件索引格式，且角色卡內嵌 `character_book` 也走同一套正規化。
- 理由：真實 SillyTavern / 修卡案例可能出現物件型 entries；若只接受陣列，拆分後再回填時容易被清成空世界書。

## 26/06/10 14:26

- 決策：角色卡主欄位改用創作者導向中文命名，並補上每個欄位的用途說明。
- 理由：使用者主要是不熟 JSON 的創作者；直接顯示 `description`、`scenario`、`first_mes` 等英文欄位名不夠直覺。

## 26/06/10 14:26

- 決策：`alternate_greetings` 改為多格逐筆編輯，而不是用 `---` 分隔的單一 textarea。
- 理由：逐格編輯更符合「多個開場首訊」的心智模型，也更方便新增、刪除與人工校對。











## 26-07-10 PNG metadata 寫入語義
- 決策：`insertCardIntoPng` 對 `chara` 與 `ccv3` 兩種 tEXt 塊一律「都清、都寫」（同一份 v3 JSON base64 寫兩塊）；`extractCardFromPng` 讀取以 ccv3 優先。
- 理由：SillyTavern 官方 write 就是雙寫、read 是 ccv3 優先（character-card-parser.js）。只清 chara 會讓封面殘留的舊 ccv3 蓋掉新資料（實測：舊卡圖當封面時 ST 讀到舊卡）。

## 26-07-10 git 歷史收編
- 決策：認 GitHub 遠端（13 commits）為正統歷史，本機孤兒歷史（3 commits）快照後 merge 收編，備份於 `backup-local-history` 分支；不做任何 force-push。
- 理由：兩邊程式碼實質相同（僅 BOM/EOL 差異），遠端 commit 訊息較完整；本機文件較新故 merge 以本機為準。部署鏈自此為 commit→push→GitHub Pages。
